## The "Personal Style"

  As an inexperienced 'code writer' (otherwise known as a maturing software engineer), it is very typical to get into personal habits when writing programs. Some may declare every variable right before they're used, while others organize every variable declaration at the top (the latter being myself). One person may view For-loops as their arch nemesis, while another has them littered everywhere throughout the page. There are countless examples of varying differences between how each person chooses to write their code, and we each have our reasons to do so. Such phenomena generally don't exhibit negative consequences when small programs are developed on an individual scale, like in an isolated bubble, but you can start to see an emerging issue when larger and more sophisticated projects include dozens or more individuals. Each developer brings their own unique stylistic choices to the table, and while technically any code that 'gets the job done' will probably suffice, anyone who has ever tried to debug a friend's project could testify that following someone else's syntax is almost equivalent to learning a new language on-the-fly.

## How do we get everyone on the same page?

  Simple. We get everybody to write with the same coding standard. Sure, there might be a slight learning curve and some old die-hard habits that has to be broken. But, as I'll soon learn this is a small price to pay for more efficient development. Its like everybody in a product's assembly line having the same blueprint for how the final assembly should be done.

  So, how does this look in practice? Well in my case, learning one specific coding style as a part of a class helped me better visualize each segment of the code with an organized and clear layout that was straightforward to follow. The benefits will not end there though. I have yet to understand the process of developing a project in conjunction with other people in a team but I could imagine how a standardized way of formatting code will dramatically expedite the process when it is handed from one person to another.

## Discovering IntelliJ, ESLint

  Utilizing an integrated development environment (IDE) is part of the process for mature software engineers. IntelliJ is an IDE that I am using that encorporates ESLint. As ChatGPT puts it, ESLint can be thought of as "your personal coding coach that points out where you might've slipped up in your coding style, potential bugs, or even areas where your code could be optimized for performance". IntelliJ and all of its many features feels overwhelming for a beginner, as if a novice pilot is asked to take control of a Boeing 747 for the first time; there are simply too many knobs and buttons that I don't know the functions of, so I'm not able to utilize the vast majority of its tools.
